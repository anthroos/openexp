## Vision

A hippocampus for AI agents. On the question "how did this happen?" the system returns the trajectory — context, hypotheses, checks, actions, outcome — backed by explicit links to real-world events (email reply, meeting attended, deal closed, payment received).

Not a memory store with magic numbers. Not a replacement for skills. A continuously growing labeled dataset of human-AI decisions tied to terminal outcomes, where labels are assigned retroactively when reality returns its verdict.

The eventual product: a model fine-tuned on this dataset that develops intuition about "how this kind of decision usually plays out" — grounded in real outcomes, not crawled internet text.

## Core Principle: No Pre-Labeling

We do NOT hand-craft features at step level (`intent=propose`, `tone=urgent`, `polarity=+1`). Pre-labeling injects the labeler's biases into the dataset and corrupts the eventual training signal.

Only terminal outcomes get labels:
- `outcome`: closed_won | closed_lost | failed | abandoned
- `grade`: 0.0–1.0, school-style

Intermediate steps are stored RAW. The model later learns from data which signals predict the grade.

Analogy: kids learn from grades and outcomes, not from a teacher annotating each homework problem with "this approach is right."

## Trajectory Schema

```yaml
trajectory:
  id: deal_fullswing
  domain: sales | dev_decisions
  started_at: 2026-02-10T00:00:00Z
  ended_at: 2026-04-22T00:00:00Z      # nullable while active

  steps:
    - ts: 2026-02-10T15:32:00Z
      kind: message | action | retrieval | decision
      session_id: <uuid>
      raw: <full text or payload, no interpretation>
      retrieved_memory_ids: [...]      # if applicable

  terminal:
    outcome: closed_won
    grade: 0.8
    grade_source: hybrid                # opus_auto | manual | hybrid
    grade_auto: 0.75                    # Opus 4.7 estimate
    grade_manual: 0.85                  # Ivan override
    grade_reason: "closed at 60% of expected price; took 3x longer than planned"
    closed_at: 2026-04-22
```

## Domains in Scope

- `sales` — outreach, proposals, negotiations, closes
- `dev_decisions` — sprint planning, architecture choices, prioritization

Out of scope for trajectory linking (transcripts still stored, just not graded): research, AGI, personal health, parenting.

## Epics

### E1 — Outcome Capture

**DoD:** `outcomes.jsonl` grows automatically from ≥3 sources without manual `log_outcome` calls.
**Sources (priority order):** Gmail reply detector → CRM stage change → Stripe payment webhook → calendar event attended.
**Experiment:** 7 days → ≥30 outcome events; manual review confirms no obvious misses.

### E2 — Trajectory Linker

**DoD:** function `trajectory_for(outcome_id)` returns ordered list of `(session_id, step refs)` within time window. Boundary detection via Opus 4.7 + system prompt; output reviewed by Ivan periodically.
**Experiment:** 10 closed deals → build trajectories → manual verification ≥80% accuracy.

### E3 — Streaming Trajectory Closure Detection

**DoD:** heuristic detects when a trajectory has stalled (N days no activity) → flag for terminal grade.
**Experiment:** 2 weeks of operation → ≥5 trajectories auto-flagged; ≥80% are real closures (not false positives).

### E4 — Hybrid Grading

**DoD:** terminal outcome triggers Opus 4.7 grading pass with system prompt, producing `(outcome, grade, reason)`. Ivan reviews + can override grade. Both stored: `grade_auto` and `grade_manual`.
**Experiment:** 10 closed deals → measure agreement between Opus grade and Ivan grade. Baseline target: |Δ| < 0.2 in 70% of cases.

### E5 — Retrieval (interim)

**DoD:** `search_memory` returns explainable ranking with per-component scores visible. Components: semantic (0.50) + BM25 (0.20) + recency (0.30). No Q-value. No provenance score.
**Experiment:** 20 queries → qualitative review.

### E6 — Legacy Removal

**Triggered AFTER E1–E5 prove themselves** (estimated 6+ weeks).
**DoD:** repo without `q_value` references in scoring path; tests updated; site + README consistent with v2 narrative.

## Pilot — Before Any Production Code

Single trajectory end-to-end, by hand:

1. Pick one closed deal (FullSwing or similar) with full Claude conversation history available
2. Manually assemble a trajectory file using the schema above
3. Run Opus 4.7 grading with system prompt → compare to Ivan's grade
4. Verify schema captures everything needed

**Time budget:** 3–4 days. If schema breaks on real data, revise before writing production code.

## Open Questions

1. **Trajectory boundary accuracy.** Opus may over-merge (one giant trajectory) or under-merge (many fragments). Calibration is part of E2.
2. **Grade calibration across domains.** Sales grade ≠ dev_decisions grade scale. May need per-domain rubrics in the system prompt.
3. **Multi-channel trajectories.** A deal spans Claude conversations, IDE sessions, manual emails, calls. What's the data fabric that ties them?
4. **Grade decay.** Should an old graded trajectory still influence retrieval at full weight, or fade with time? (Defer — semantic+recency only for now.)
